# Retrospect — セッション知見の永続化

セッションで得た知見を「次のセッションに持ち越せる形」で保存する。何も保存せずにセッションを終えることは知見の損失である。

## 手順

### 1. セッションを振り返り、4種類の知見を抽出する

- **(a) 環境・設定の事実**: このマシン・リポジトリ・ツールについて新たに判明したこと（例: 「jq は /usr/bin に標準で入っている」）
- **(b) ユーザーからの修正・好み**: 作業方針への指摘、選好、確立した規約（why を必ず添える）
- **(c) 技術的な学び**: 一般化できる技術知見（特定ツールの挙動、設計パターン、落とし穴）
- **(d) 未解決事項・次の一手**: やり残し、保留した判断

### 2. 保存先に振り分ける

| 種類 | 保存先 | 形式 |
|---|---|---|
| (a) 環境・事実 | Claude Code memory（`~/.claude/projects/<project>/memory/`） | frontmatter 付き1ファイル1事実、`MEMORY.md` の index も更新 |
| (b) 修正・好み | 同上（`type: feedback`） | **Why:** と **How to apply:** を必ず書く |
| (c) 技術知見 | `~/works/knowledge-clawler/inbox/` | `templates/source-capture.md` 準拠。`source_type: experiment`、`research_prompt` にセッション文脈を記載。vault 側の /reduce が後で処理する |
| (d) 未解決事項 | 該当プロジェクトの管理ファイル（knowledge-clawler なら `ops/goals.md`） | 既存形式に従う |

### 3. 保存前の品質チェック

- 既存 memory と重複していないか（既存ファイルの更新を優先、矛盾したら古い方を直す）
- リポジトリや git 履歴から分かることは保存しない（コード構造・過去の修正内容は対象外）
- (c) は命題形のタイトルにできるか（「〜は〜である」と主張できないものは知見ではない）

### 4. 報告

保存した知見を保存先ごとに1行ずつ列挙して報告する。保存すべきものが無かった場合は「新規知見なし」と明言する。
